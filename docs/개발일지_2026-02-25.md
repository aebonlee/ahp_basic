### 2. [P0] ToastContext useCallback → useMemo 버그 수정

**문제**: `useCallback`은 함수 전용 메모이제이션 훅인데, 객체에 적용하고 있었음. React API 오용.

```javascript
// Before (잘못된 사용)
const toast = useCallback({
  success: (msg) => addToast(msg, 'success'),
  error: (msg) => addToast(msg, 'error'),
}, [addToast]);

// After (올바른 사용)
const toast = useMemo(() => ({
  success: (msg) => addToast(msg, 'success'),
  error: (msg) => addToast(msg, 'error'),
}), [addToast]);
```

**수정 파일**: `src/contexts/ToastContext.jsx`

### 3. [P0] 관리자 권한 DB 기반 전환

번들에 포함되어 전세계에 공개되며, 관리자 추가 시 코드 수정+재배포 필요.

**수정 내용**:
- `ADMIN_EMAILS` 배열 완전 제거
- `user_profiles.role === 'admin'` DB 컬럼 기반으로 전환
- `profileLoading` 상태 추가 — 프로필 로드 완료 전 잘못된 리디렉션 방지

```javascript
// Before
const ADMIN_EMAILS = ['aebon@kakao.com', ...];
const isAdmin = isLoggedIn && ADMIN_EMAILS.includes(state.user?.email);

// After
const isAdmin = isLoggedIn && state.profile?.role === 'admin';
```

**수정 파일**: `src/contexts/AuthContext.jsx`

## 모델 구축 드래그 앤 드롭 + 더블클릭 인라인 편집 (세션 6)

### 26. 드래그 앤 드롭 순서 변경 / 부모 변경

**배경**: 모델 구축 페이지에서 기준·대안의 순서를 변경하거나 계층 구조를 수정하려면 삭제 후 재생성해야 했음. 마우스 드래그로 직관적 조작이 불가능.

**구현 내용**:

| 기능 | 설명 |
|------|------|
| **형제 간 순서 변경** | 같은 부모의 기준/대안 노드를 드래그하여 앞·뒤로 이동 |
| **부모 변경 (re-parent)** | 기준 노드를 다른 기준 노드 가운데에 드롭하면 하위 기준으로 이동 |
| **최상위로 이동** | 기준 노드를 목표(Goal) 노드에 드롭하면 루트 레벨로 이동 |
| **대안 순서 변경** | 대안 노드 간 드래그로 순서 변경 |
| **순환 참조 방지** | 자기 자신 또는 하위 노드에 드롭 불가 (descendantIds 체크) |
| **드래그 취소** | Escape 키로 드래그 취소 |

**드롭 존 판정 (기준 노드)**:
- 상단 30%: `before` → 해당 노드 앞에 삽입
- 하단 30%: `after` → 해당 노드 뒤에 삽입
- 중앙 40%: `child` → 해당 노드의 하위 기준으로 이동

**시각적 피드백**:
- 고스트 노드: 드래그 중 반투명 복제본이 마우스를 따라감
- 원본 노드: opacity 0.25로 페이드
- 드롭 인디케이터: before/after는 파란 삽입선, child는 점선 테두리

**핵심 로직** (`useCriteria.moveCriterion`):
1. 대상 부모 레벨의 형제 목록 재구성
2. 지정 인덱스에 이동 항목 삽입
3. 전체 형제의 `sort_order` 재계산
4. 부모가 변경된 경우 이전 부모의 형제도 재인덱싱
5. `Promise.all`로 DB 배치 업데이트

### 38. 설계 원칙

**문제**: 기존 ModeSwitch는 "관리자 MODE" / "평가자 MODE로 이동" 라벨로, 모든 사용자(연구자·평가자)에게 동일하게 표시됨. 실제 평가자가 "관리자 MODE로 이동"을 클릭하면 AdminGuard에 의해 차단되지만, 혼란을 유발. 연구자가 평가자 화면을 테스트하려는 의도도 명확하지 않음.

## 가로형 캔버스 노드 동적 높이 + 레벨별 통일 (세션 11)

### 47. 가로형 노드 텍스트 줄바꿈 + 높이 동적 계산

**문제**: 가로형(horizontal) 모드에서 모든 노드 높이가 `H_NODE_HEIGHT = 56px`로 고정. 긴 텍스트는 `-webkit-line-clamp: 3`으로 3줄까지만 표시되고 나머지 잘림. 같은 레벨에서 짧은 텍스트와 긴 텍스트의 노드 높이가 동일하지 않아(실제로는 동일 고정값이지만 텍스트가 잘림) 시각적 일관성 부족.

**해결 방식**:
- 텍스트 줄 수를 사전 추정(`estimateTextLines`) → 필요 높이 계산(`estimateNodeHeight`)
- 같은 레벨의 모든 노드가 가장 높은 노드 기준으로 높이 통일(`levelMaxHeight`)
- 대안 노드도 별도 레벨로 높이 통일 적용

**추가 함수** (`src/lib/hierarchyLayout.js`):

```javascript
function estimateTextLines(label, nodeWidth) {
  // 노드 폭에서 패딩 제외한 텍스트 영역 기준으로 줄 수 추정
  // 한글 ~11px, 영문 ~7px (0.82rem 기준)
}

function estimateNodeHeight(label, nodeWidth) {
  const lines = estimateTextLines(label, nodeWidth);
  const lineHeightPx = 17; // ~0.82rem × 1.3
  return Math.max(H_NODE_HEIGHT, lines * lineHeightPx + 20);
}
```

**`computeHorizontalLayout` 변경점**:

| 단계 | Before | After |
|------|--------|-------|
| Step 2 | 없음 | `measureHeights()` — 레벨별 최대 높이 수집 |
| 높이 참조 | `nodeHeight` (고정 56px) | `getNodeHeight(level)` (레벨별 동적) |
| subtreeHeight | `nodeHeight` 사용 | `getNodeHeight(node.level)` 사용 |
| 대안 높이 | `nodeHeight` | `altNodeHeight = getNodeHeight(altLevelKey)` |
